# Project Manual: IMDb Search System Using Local MongoDB

## 1. Project Overview

This project will build a local IMDb search system using **MongoDB** as the database. The system will use selected IMDb dataset files, import the data into a local MongoDB database, create searchable movie/title documents, and provide a simple user interface where a user can search IMDb records.

The user interface will allow users to select what they are looking for, choose filters such as title, genre, year, rating, and title type, then choose how many results they want to retrieve. The result range will be between **10 and 200 records**.

The project should first be developed using a **small sample dataset** so that UI development and testing are fast. After the UI and search logic work correctly, the same system can be tested with a larger IMDb dataset.

The main focus of the project is not only searching IMDb data. The main focus is to demonstrate:

* local MongoDB database setup,
* document-based data modeling,
* importing real-world dataset files,
* data cleaning,
* MongoDB querying,
* indexing,
* result limiting,
* search optimization,
* and performance comparison.

IMDb provides non-commercial datasets as gzipped TSV files and allows local copies for personal and non-commercial use under their license terms. The dataset files use headers in the first row and use `\N` to represent missing values. ([IMDB Developer][1])

# 10. MongoDB Document Design

Each document represents one IMDb title.

## 10.1 Example Document

```text
{
  tconst: "tt0111161",
  titleType: "movie",
  primaryTitle: "The Shawshank Redemption",
  originalTitle: "The Shawshank Redemption",
  searchTitle: "the shawshank redemption",
  isAdult: false,
  startYear: 1994,
  endYear: null,
  runtimeMinutes: 142,
  genres: ["Drama"],
  rating: {
    averageRating: 9.3,
    numVotes: 2950000
  },
  importedAt: timestamp
}
```

## 10.2 Field Explanation

| Field                  | Type             | Purpose                                      |
| ---------------------- | ---------------- | -------------------------------------------- |
| `tconst`               | string           | IMDb unique title ID                         |
| `titleType`            | string           | movie, tvSeries, short, video, etc.          |
| `primaryTitle`         | string           | main title shown to users                    |
| `originalTitle`        | string           | original title                               |
| `searchTitle`          | string           | lowercase version of title for easier search |
| `isAdult`              | boolean          | adult content flag                           |
| `startYear`            | integer or null  | release/start year                           |
| `endYear`              | integer or null  | end year for TV series                       |
| `runtimeMinutes`       | integer or null  | runtime                                      |
| `genres`               | array of strings | searchable genre list                        |
| `rating.averageRating` | float or null    | IMDb average rating                          |
| `rating.numVotes`      | integer or null  | number of votes                              |
| `importedAt`           | date/time        | when the record was imported                 |

## 10.3 Why Embed Rating Inside Title?

The IMDb source files are separate, but MongoDB allows related data to be embedded.

The app usually displays title and rating together.

So instead of storing ratings in a separate collection, embed rating inside each title document:

```text
rating.averageRating
rating.numVotes
```

This avoids needing a join-like operation during search.

## 10.4 Why Store Genres as an Array?

IMDb genres are comma-separated in the source file.

Example:

```text
Action,Crime,Drama
```

In MongoDB, store this as:

```text
["Action", "Crime", "Drama"]
```

This allows simple genre filtering.

Example user filter:

```text
Genre = Action
```

MongoDB can match documents where the `genres` array contains `"Action"`.

MongoDB array fields can also be indexed; MongoDB treats this as a multikey index.

# 11. Data Cleaning Rules

Data cleaning is essential because IMDb source files store all values as text.

The import process must convert values into useful database types.

| IMDb Raw Value      | Stored Value                |
| ------------------- | --------------------------- |
| `\N`                | null                        |
| `"1994"`            | integer `1994`              |
| `"142"`             | integer `142`               |
| `"9.3"`             | float `9.3`                 |
| `"2950000"`         | integer `2950000`           |
| `"Action,Drama"`    | array `["Action", "Drama"]` |
| `"0"` for `isAdult` | false                       |
| `"1"` for `isAdult` | true                        |

## 11.1 Missing Values

IMDb uses:

```text
\N
```

for missing values.

These should never be stored as the literal string `"\N"`.

They should be converted to:

```text
null
```

This matters because filtering numeric fields will not work correctly if missing values are stored as strings.

## 11.2 Year Conversion

`startYear` and `endYear` should be integers when present.

Bad:

```text
"2005"
```

Good:

```text
2005
```

## 11.3 Rating Conversion

`averageRating` should be a float.

Bad:

```text
"8.2"
```

Good:

```text
8.2
```

## 11.4 Votes Conversion

`numVotes` should be an integer.

Bad:

```text
"1500000"
```

Good:

```text
1500000
```

## 11.5 Genre Conversion

If the raw genre value is:

```text
Action,Crime,Drama
```

store it as:

```text
["Action", "Crime", "Drama"]
```

If the raw value is missing:

```text
\N
```

store it as:

```text
[]
```

or:

```text
null
```

Recommended choice:

```text
[]
```

because it makes genre filtering easier and avoids null-related checks.

# 12. Data Import Workflow

The import process should follow this exact sequence.

## Step 1: Start Local MongoDB

Before importing, local MongoDB must be running.

The database server should be available at:

```text
mongodb://localhost:27017/
```

## Step 2: Connect to MongoDB

Connect to:

```text
Database: imdb_search
Collection: titles_sample or titles_full
```

## Step 3: Clear Existing Collection

During development, the import script should optionally clear the selected collection before inserting new data.

This prevents duplicate records.

## Step 4: Read Ratings File First

Read `title.ratings.tsv.gz`.

Build a lookup structure using `tconst`.

Conceptually:

```text
rating_lookup[tconst] = {
  averageRating,
  numVotes
}
```

Reason:

When reading title records later, the import process can quickly attach rating information.

## Step 5: Read Title Basics File

Read `title.basics.tsv.gz` row by row.

For each row:

1. Read `tconst`.
2. Check whether this title has a rating.
3. Clean title fields.
4. Convert data types.
5. Convert genre string into array.
6. Attach rating object.
7. Build MongoDB document.
8. Insert into collection.

## Step 6: Apply Development Filters

For sample UI development, do not import everything.

Use filters such as:

```text
titleType = movie
rating exists
startYear is not missing
numVotes >= 1000
```

This creates a cleaner sample collection.

Why?

Because if the first 10,000 IMDb rows are imported directly, many may be old, obscure, unrated, or not useful for UI testing.

A better sample should include searchable and recognizable records.

## Step 7: Apply Record Limit

For sample mode:

```text
Maximum records = 10,000
```

For larger mode:

```text
Maximum records = 100,000 or more
```

## Step 8: Insert in Batches

For performance, insert documents in batches rather than one by one.

Recommended batch size:

```text
500 to 5000 documents per batch
```

This makes import faster and reduces database overhead.

## Step 9: Print Import Summary

At the end, print:

```text
Collection name
Total rows read
Total ratings loaded
Total documents inserted
Total skipped records
Import duration
```

# 13. Search Requirements

## 13.1 Keyword Search

```text

```

```text
primaryTitle
```

Recommended behavior:

* case-insensitive matching,
* support partial matching,
* allow empty keyword.

Important note: normal indexes help regular expression searches mainly when the regex is prefix-based. MongoDB documentation explains that prefix expressions can use an index more effectively because MongoDB can construct a range from that prefix. ([MongoDB][5])

So the project should explain this limitation:

> A title contains-search such as “contains Batman” may not always benefit from a normal index. A prefix search such as titles starting with “Batman” is more index-friendly.

# 16. Indexing Requirements

Indexing is a major part of the project.

The team must create indexes after importing data.

## 16.1 Required Single-Field Indexes

Create indexes on:

```text
tconst
primaryTitle
searchTitle
titleType
startYear
genres
rating.averageRating
rating.numVotes
```

## 16.2 Required Compound Indexes

Create compound indexes for common combined searches.

Recommended compound indexes:

```text
titleType + startYear
```

Purpose:

* filtering movies by year range.

```text
genres + startYear + rating.averageRating
```

Purpose:

* filtering genre and year, then sorting/filtering by rating.

```text
titleType + genres + startYear + rating.averageRating
```

Purpose:

* combined movie search with title type, genre, year, and rating.

MongoDB recommends thinking carefully about compound index order. The Equality, Sort, Range guideline says equality fields usually come first, followed by sort and range fields depending on query needs. ([MongoDB][6])

## 16.3 Indexing Explanation for Report

The report should explain:

* `titleType` is used for equality filtering,
* `genres` is used for equality filtering inside an array,
* `startYear` is used for range filtering,
* `rating.averageRating` is used for sorting/filtering,
* compound indexes help multi-condition queries.

## 16.4 Text Search Option

MongoDB has text indexes for `$text` queries on string fields, and a collection can have one text index that may cover multiple fields. ([MongoDB][7])

However, since the course appears to focus more on general MongoDB indexes, text search should be optional.

Recommended decision:

* Use regular indexes for the main project.
* Mention text index as future improvement or optional extension.
* Do not depend on Atlas Search because the project is local MongoDB.

# 23. Important Implementation Notes

## 23.1 Do Not Import Full Data First

The full dataset may slow down development.

Start with:

```text
titles_sample
```

Only after the UI works, use:

```text
titles_full
```

## 23.2 Validate Data Types

MongoDB queries depend on correct data types.

If `startYear` is stored as string, range queries may behave incorrectly.

Correct:

```text
startYear: 2005
```

Incorrect:

```text
startYear: "2005"
```

## 23.3 Do Not Store `\N`

No field should contain raw `\N`.

Convert it to null or empty array.

## 23.4 Always Limit Results

The UI must always use a result limit.

Never return the whole collection.

## 23.5 Index After Import

Indexes should usually be created after bulk import.

Reason:

* importing into an indexed collection can be slower,
* creating indexes after import is cleaner for this project.

## 23.6 Be Careful with Regex Search

Title keyword search using regular expressions may not always use indexes efficiently.

Prefix-style regex can be more index-friendly than contains-style regex. MongoDB documentation specifically notes that prefix expressions can use indexes more effectively. ([MongoDB][5])

For the report, explain:

> Genre, year, rating, and title type filters benefit more clearly from indexes. Title keyword search with contains-style regex is useful for the UI but may not always show large index improvement.

## 23.7 Use Explain for Database-Focused Analysis

The benchmark should not rely only on time.

Use explain summaries to show:

```text
COLLSCAN before index
IXSCAN after index
documents examined reduced
execution time reduced
```
